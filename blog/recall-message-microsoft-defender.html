Remove a delivered email using Microsoft Defender

This guide covers how to remove a delivered message across mailboxes using Microsoft Defender for Office 365: how to scope the message correctly, run a remediation action, handle AIR approvals, and verify completion.

Updated: 2026-03-07 Audience: Security ops / M365 admins Applies to: Defender for Office 365
Important distinction Outlook “recall” is a client-side experience and is not a reliable control for enterprise remediation. Use Defender workflows to remove or neutralize the message at the service level.

Prerequisites

  • Microsoft Defender for Office 365 Plan 1 or Plan 2 (features vary by license).
  • Permissions to access Defender portal and run remediation actions.
  • At least one stable message identifier (preferred: Internet Message ID).

Step 1: Identify the message deterministically

Use stable identifiers to prevent removing the wrong message. Best options, in order:

  • Internet Message ID (globally unique for the message instance)
  • Message ID + sender + recipient + time window
  • Subject alone is not sufficient (subject collisions are common)
Do not guess If you cannot uniquely identify the message, widen the investigation first. Wrong remediation is worse than delayed remediation.

Step 2: Scope in Explorer

In Microsoft Defender portal, use Explorer to confirm:

  • Which recipients received the message
  • Delivery status (delivered, quarantined, blocked)
  • Current location (Inbox/Junk/Deleted Items)
  • User interaction (clicked URL, opened attachment) if available

Step 3: Run remediation action

Choose a remediation action appropriate for the incident severity. For confirmed malicious content, remove it. For uncertain cases, use a containment-first approach and continue investigation.

AIR approvals and status

Automated Investigation and Response (AIR) can create actions that require approval depending on your configuration. A “pending” action is not a remediation. Remediation is complete only when the action shows completed successfully.

Step 4: Verify completion

Verification should include both workflow status and evidence in Explorer:

  • Action status is completed (not pending)
  • Explorer no longer shows the message as delivered/available in mailboxes
  • For targeted mailboxes, confirm the message is removed or moved as expected

If removal does not complete

Common reasons remediation “does nothing”:

  • Message was not uniquely identified and the action targeted the wrong artifact
  • Action requires approval and is still pending
  • Mail was delivered in variants (forwarded, resent) and you targeted only one instance
  • Retention/hold or mailbox constraints prevent expected movement behavior (rare, but happens)
