2026-03-02 Entra ID Authentication AADSTS

How to resolve AADSTS76026: “The request has expired. Try to submit new request.”

This error almost always means one thing: the sign-in request you are trying to complete is no longer valid. Not “your tenant is broken”, not “Microsoft is down”, just a stale auth request.

What it means

Entra ID (Azure AD) uses short-lived values during interactive authentication (for example, an authorization code, state, nonce). If the user tries to finish a sign-in using an old browser tab, an old link, or a cached page, Entra refuses it. That refusal is what shows up as AADSTS76026.

Fast fix checklist (do this first)

  • Close all tabs for the app, M365 portal, and login.microsoftonline.com.
  • Open a fresh InPrivate/Incognito window and try the sign-in again.
  • Do not use the browser back button during the sign-in flow.
  • If it is a password reset or MFA registration flow, restart the process from the original app entry point.
  • Verify your device time is correct (clock skew can break auth flows).

Common root causes

  • Stale sign-in link: user clicked an old email link, old bookmark, or reopened yesterday’s tab.
  • Back button replay: user completed part of the flow, then navigated back and resubmitted.
  • Proxy or gateway caching: a reverse proxy caches auth pages and serves old content.
  • Long user delays: user started sign-in and waited too long before completing MFA or consent.
  • Clock skew: device time is off by a few minutes.
  • Multiple sessions: user is signed into multiple accounts and the browser mixes cookies.

Admin checks (when the fast fix does not work)

These checks help you decide if this is a user behavior issue or a real configuration issue.

  • Sign-in logs: check the sign-in entry and confirm the failure is AADSTS76026.
  • Conditional Access: confirm there is no policy forcing repeated reauthentication loops (Sign-in frequency, device compliance).
  • Client type: see if it is happening in browser only, Outlook only, or a specific app.
  • App configuration: for custom apps, verify redirect URI and that the app is not reusing old authorization responses.
Tip for stubborn cases: If only one app is affected and Incognito does not help, suspect caching or session handling in that app or its proxy. If many apps are affected for the same user, suspect browser cookie chaos or device time.

Prevent it in the future

  • Train users: “If sign-in fails, close the tab and start again.”
  • Avoid sending “deep” auth links that expire quickly.
  • On reverse proxies, disable caching for auth endpoints and callback URLs.
  • For custom apps, ensure you never reuse an authorization code or replay an old state/nonce.
