AADSTS76026: The request has expired

Fix AADSTS76026 for SAML apps that fail when launched from My Apps. Set the Sign-on URL so Entra starts a fresh request, then verify in sign-in logs.

2026-03-07 8 min read Entra ID SAML
← Back to Blog

You might see AADSTS76026: The request has expired. Try to submit new request. when users launch a SAML app from My Apps, but the same app works when users start from the application’s own sign-in page.

Applies to Microsoft Entra ID (Enterprise applications) using SAML single sign-on.

Why this error happens

In a SAML flow, the application (service provider) generates a sign-in request that includes a timestamp. Entra ID validates that timestamp to help prevent replay. If Entra receives an older request, it rejects it as expired.

This commonly happens when users start the sign-in from My Apps and the enterprise app doesn’t have a valid entry point. Without the correct entry point, the app can reuse a stale request or fail to generate a new request at the right time.

Fix: Set the Sign-on URL in the SAML configuration

Set the Sign-on URL so My Apps launches redirect users to the application’s SAML entry page (SP-initiated URL). This helps the application generate a fresh SAML request.

Steps

In the Entra admin center:

1. Go to Identity > Applications > Enterprise applications.
2. Select the application.
3. Select Single sign-on > SAML.
4. Under Basic SAML Configuration, select Edit.
5. In Sign-on URL, enter the application’s SP-initiated sign-in URL (the URL users normally use to start SSO from the app).
6. Select Save.

Tip If you are not sure what URL to use, start with the application’s standard sign-in page that triggers SSO. Many vendors document this as the “SP-initiated SSO URL” or “SSO start URL.”

Verify the fix

Use a clean browser session so you don’t validate against cached redirects.

1. Open an InPrivate/Incognito window.
2. Go to myapps.microsoft.com.
3. Select the application tile.
4. Confirm the sign-in completes without AADSTS76026.

If the issue continues

If the Sign-on URL is correct and the error continues, focus on time consistency and request replay.

Check for clock skew

If the application server (or an upstream component) has incorrect time, Entra can treat the request as expired. Ensure app servers and any federation/proxy components use a reliable NTP source.

Check for proxy or gateway replay

If a reverse proxy, WAF, or gateway caches authentication pages or redirects, it can cause reuse of an old request. Review caching rules and disable caching for sign-in endpoints.

Confirm the pattern in sign-in logs

In Entra admin center, use Sign-in logs to compare: a failed My Apps launch versus a successful app-initiated launch. This helps confirm the issue is launch-path specific.
