Microsoft Entra ID · SAML single sign-on

AADSTS76026: The request has expired

This error indicates that Microsoft Entra ID received a SAML request outside the allowed time window. The most common cause in production is an incorrect launch entry point (My Apps tile launching without a valid Sign-on URL), which results in stale or replayed requests.

Updated: 2026-03-07 Audience: Entra / IAM admins Applies to: Enterprise apps (SAML)
At a glance If this error occurs only when launching from myapps.microsoft.com, set the application Sign-on URL in the SAML configuration so the service provider generates a fresh request at launch.

Symptoms

Typical patterns seen in sign-in incidents:

  • Users receive AADSTS76026 when clicking the app tile in My Apps.
  • The same user can often sign in successfully when starting from the application’s own sign-in page.
  • Failures may cluster around specific browsers or networks if caching or proxy behavior is involved.

Scope and prerequisites

This guidance assumes:

  • The enterprise application is configured for SAML single sign-on.
  • You have permission to edit the enterprise app SSO configuration in the Entra admin center.
  • You can review Sign-in logs for the affected application.

How SAML requests expire

In an SP-initiated SAML flow, the service provider (application) generates a SAML authentication request. That request typically includes a timestamp and may also be bound to a nonce or request ID. Microsoft Entra ID validates that the request is recent enough to reduce replay.

When Entra ID receives a request that is outside the allowed window, it returns AADSTS76026. That window can be exceeded by:

  • Requests being generated too early and reused later (stale request).
  • Redirect chains being cached or replayed by an intermediary (proxy/WAF/caching layer).
  • System time skew on the SP side (application servers not NTP-synced).

Root cause (most common in My Apps launches)

When a user launches an app from My Apps, Entra needs an explicit entry point that tells it where to send the user to begin SP-initiated SSO. If the enterprise application’s SAML configuration does not have a correct Sign-on URL (or it points to a non-SSO endpoint), My Apps launches can lead to a stale request pattern:

  • The user reaches the application in a way that does not trigger fresh SAML request generation.
  • The app (or an intermediary) replays an older request.
  • Entra rejects it as expired.
Important Don’t “fix” this by relaxing security controls in the identity provider. Fix the launch path so the service provider creates the request at the time of launch.

Fix (deterministic)

Set the Sign-on URL in the enterprise application’s SAML configuration to the application’s SSO start URL (often documented by the vendor as “SP-initiated SSO URL” or “SSO start URL”).

Steps

  1. Open Microsoft Entra admin center.
  2. Go to Enterprise applications → select your application.
  3. Open Single sign-on → select SAML.
  4. In Basic SAML Configuration, select Edit.
  5. Set Sign-on URL to the application’s SSO start URL.
  6. Select Save.
What to use for Sign-on URL Use the vendor’s documented SP-initiated entry point. It is not always the same as the application’s generic login page.

Verification

After saving the configuration, verify using a clean session to avoid cached redirects:

  1. Open an InPrivate/Incognito browser window.
  2. Sign in to https://myapps.microsoft.com.
  3. Launch the application tile.
  4. Confirm the sign-in completes without AADSTS76026.

Use sign-in logs to confirm the mechanism

In the Entra admin center, review sign-in logs for the application and compare:

  • A failed My Apps launch (before change)
  • A successful My Apps launch (after change)

Look for differences in the request timing and the initiating application context.

If the issue continues

If the correct Sign-on URL is set and the error persists, troubleshoot in this order:

1) Time skew on service provider infrastructure

Ensure application servers and any identity gateway components have stable time synchronization (NTP). Time drift on the SP side is a direct cause of expired SAML requests.

2) Proxy/WAF/caching replay behavior

Check whether a reverse proxy, WAF, or edge cache is caching redirects or replaying the same request. Authentication flows must not be cached.

3) Session reuse / stuck browser state

Validate from a clean browser profile. Some failures are driven by old cookies, extensions, or SSO plugins interfering with redirect chains.
