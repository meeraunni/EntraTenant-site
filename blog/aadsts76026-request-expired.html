AADSTS76026: The request has expired

This error is almost never “a random outage.” It’s a request validity problem in the SAML authentication pipeline. The fix depends on where the request is generated and whether the flow is IdP-initiated or SP-initiated.

2026-03-038 min readSAMLAADSTS
Engineer translation: Entra received a SAML authentication request whose timestamp window was no longer acceptable (or the request was effectively replayed). The platform returns AADSTS76026 (RequestIssueTimeExpired) when the request issue time is outside the allowed validity window.

How this breaks in real architectures

In SAML, the service provider (the application) generates an AuthnRequest. That request contains time-bound fields (IssueInstant / IssueTime), and the IdP (Entra ID) validates freshness to prevent replay. If the request is old, duplicated, cached by an intermediary, or generated from an entry point that reuses a previous request, the IdP rejects it.

The pattern you described is a common one: SP-initiated login works when users start from the application, but IdP-initiated login fails when users launch the app from My Apps. In IdP-initiated mode, Entra needs a stable entry URL to send users into the app’s SAML start logic so the app generates a fresh request every time. If that entry point is missing or wrong, you get stale or malformed requests and the expiry shows up as AADSTS76026.

Validation: prove the failure mode instead of guessing

Start with Entra sign-in logs and treat them like a trace. Filter by the user and the application, then open a failure and capture three things: the protocol (SAML), the client (browser vs brokered client), and the exact error code. You are checking whether failures correlate with IdP-initiated launches (My Apps tile) versus SP-initiated launches (application URL).

The second validation is time consistency. If the application server or any front-door component generating requests has clock skew, the IssueInstant can be interpreted as “already expired.” This is less common than the My Apps entry-point issue, but when it happens it is brutal because it looks intermittent and user-specific.

Remediation: set the Sign-on URL for IdP-initiated launches

For SAML enterprise apps in Entra, the Sign-on URL is the entry point Entra can use to launch the app. When users click the tile in My Apps, Entra can redirect them to this URL so the app runs its SP-initiated SAML logic and generates a fresh AuthnRequest. That avoids stale requests and stops the “request expired” loop.

Implementation in Entra admin center: go to Enterprise applications, open the app, select Single sign-on, choose SAML, edit Basic SAML Configuration, and set Sign-on URL to the application’s SP-initiated login/start endpoint (the URL the app expects users to hit to start SSO). Save, then test from an InPrivate session via myapps.microsoft.com.

Hardening: stop this from returning

Treat this as a design problem, not a user problem. Ensure the application has a single canonical SP-initiated entry URL and that My Apps launches route into it. Validate that no reverse proxy or WAF is caching authentication requests or redirect responses. Ensure NTP is correct on the app tier. Finally, document the supported launch paths so helpdesk doesn’t distribute a “deep link” that fails later due to validity windows.
