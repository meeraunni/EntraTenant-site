2026-03-03 Entra ID SAML SSO AADSTS76026

AADSTS76026: The request has expired. Try to submit new request.

This error is a SAML timing problem. Microsoft maps AADSTS76026 to RequestIssueTimeExpired, meaning the IssueTime inside the SAML2 authentication request is expired. :contentReference[oaicite:2]{index=2}

The “real-world” symptom: It often happens when users launch the app from My Apps (IdP-initiated) and the app configuration is missing a proper entry point. The fix that usually stops it: set the Sign-on URL in the app’s SAML configuration.

What this error means

In SAML SSO, the service provider (your app) generates a SAML AuthnRequest that includes a timestamp (IssueInstant / IssueTime). Entra validates that timestamp. If the request looks “too old” (or gets replayed), Entra rejects it as expired. :contentReference[oaicite:3]{index=3}

Common scenario: it works SP-initiated, fails from My Apps

You’ll see this pattern a lot:

  • SP-initiated login (starting from the app) works.
  • IdP-initiated login (user clicks the tile in My Apps) fails with AADSTS76026.

When users click the tile in My Apps, Entra needs a reliable entry point to initiate the sign-in flow. That’s where the Sign-on URL comes in. The Sign-on URL is the application entry point used for user-initiated launches from Entra (including My Apps). :contentReference[oaicite:4]{index=4}

Main fix: specify the Sign-on URL in Entra SAML configuration

In the Enterprise Application:

  1. Go to Entra admin center → Enterprise applications.
  2. Select your application.
  3. Go to Single sign-on → choose SAML.
  4. In Basic SAML Configuration, click Edit.
  5. Set Sign on URL to your app’s SP-initiated login URL (the page that starts SAML login).
  6. Save.
What should the Sign-on URL be?
It should be the URL where your application starts the SAML login flow (your “login” or “SSO” landing endpoint). Microsoft’s SAML setup guidance calls this the SP Initiated Login URL value. :contentReference[oaicite:5]{index=5}

Why this resolves AADSTS76026 in this scenario: clicking the app tile now sends the user to a proper entry point that generates a fresh AuthnRequest, instead of a stale/replayed request flow.

Verify the fix

  1. Open a new InPrivate window (fresh cookies).
  2. Go to myapps.microsoft.com.
  3. Click your app tile.
  4. You should be redirected to the Sign-on URL and then signed in successfully.

Optional: validate in sign-in logs

  • Entra admin center → Sign-in logs
  • Find the failed attempt timestamp
  • Confirm the error stops occurring after the change

Other causes to check (if Sign-on URL doesn’t fix it)

  • Clock skew on the server generating AuthnRequests (or on the user device) can make IssueInstant look “too old”. :contentReference[oaicite:6]{index=6}
  • Proxy or caching layer accidentally replaying an auth request.
  • Signed request validation issues can surface as timing-related failures when signatures are validated against strict windows (common in SAML).
Practical rule: If it fails only from My Apps, start with Sign-on URL. If it fails everywhere, check time sync and replay/caching behavior first.

Wrap up

AADSTS76026 is an “expired request” error in SAML, not a mysterious tenant curse. :contentReference[oaicite:7]{index=7} If your app works SP-initiated but fails from My Apps, setting the Enterprise App’s Sign-on URL is the first fix to apply. :contentReference[oaicite:8]{index=8}
