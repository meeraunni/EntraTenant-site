You don’t have permission to send messages from this mailbox

This error occurs when Exchange Online rejects a delegated send attempt. The root cause is almost always a mismatch between the From identity Outlook is using and the actual permission model configured (Send As vs Send on Behalf), or a client using stale cached authorization state after a permission change.

Updated: 2026-03-07 Applies to: Shared mailboxes and delegated send Scope: Exchange Online
Fast triage rule Test the send action in Outlook on the web first. If OWA works but Outlook desktop fails, you are likely dealing with client-side caching. If OWA fails, you have a server-side permission or configuration issue.

Prerequisites

  • Exchange Online admin access (or the ability to run Exchange Online PowerShell).
  • A clear target identity: which mailbox the user is attempting to send as/on behalf of.
  • Clarity on intended behavior: Send As or Send on Behalf.

Understand the send model

Delegated send is not a single permission. The three commonly confused permissions are:

  • Full Access: allows opening the mailbox. Does not grant sending rights.
  • Send As: the message appears as if it was sent by the mailbox identity.
  • Send on Behalf: the message appears as “User on behalf of Mailbox”.
Common production mistake Assigning only Full Access and expecting send to work. Full Access is not send authorization.

Root cause categories

1) Missing or incorrect send permission

The user is attempting to send using an identity that Exchange Online will not authorize. This includes:

  • Send As is required, but only Send on Behalf is configured (or vice versa).
  • Permission is granted to a group that is not supported in the path you are using.
  • Permission exists, but was removed/re-added and the client is still using stale state.

2) Permission propagation delay

Some permission changes can take time to fully propagate. During that window, OWA may behave differently than Outlook desktop. Treat this like configuration drift until verified.

3) Outlook desktop caching and profile state

Outlook desktop caches certain aspects of mailbox access and sender resolution. After permission changes, it can continue to fail until the cached state refreshes. This is why OWA is the correct control test.

Deterministic fix workflow

Use this sequence to eliminate guesswork:

  1. Confirm the intended send behavior (Send As vs Send on Behalf).
  2. Validate server-side permissions in Exchange Online.
  3. Test in OWA (server-side behavior).
  4. If OWA works, remediate Outlook desktop caching/profile state.

Step 1: Validate permissions (Exchange Online PowerShell)

Replace the values below with your mailbox and user. Run these queries to confirm the effective permissions. 

# Connect-ExchangeOnline

$Mailbox = "shared@domain.com"
$User    = "user@domain.com"

# Full Access (mailbox open)
Get-MailboxPermission -Identity $Mailbox |
  Where-Object { $_.User -like "*$User*" -and $_.IsInherited -eq $false } |
  Format-Table User,AccessRights,Deny,IsInherited

# Send As (recipient permission model)
Get-RecipientPermission -Identity $Mailbox |
  Where-Object { $_.Trustee -like "*$User*" } |
  Format-Table Trustee,AccessRights,IsInherited

# Send on Behalf (mailbox property)
(Get-Mailbox -Identity $Mailbox).GrantSendOnBehalfTo

Step 2: Assign the correct permission

Option A: Send As

Use Send As when you want the message to appear from the shared mailbox identity only. 

Add-RecipientPermission -Identity $Mailbox -Trustee $User -AccessRights SendAs -Confirm:$false

Option B: Send on Behalf

Use Send on Behalf when you want attribution visible (“User on behalf of Mailbox”). 

Set-Mailbox -Identity $Mailbox -GrantSendOnBehalfTo @{Add=$User}

Step 3: Verify in Outlook on the web (OWA)

OWA is the cleanest verification step because it minimizes client caching variables:

  1. Sign in to OWA.
  2. Open the shared mailbox (Open another mailbox or direct URL if you use it).
  3. Compose a message and set the From to the shared mailbox.
  4. Send to an internal test recipient.

If OWA fails with the same error, the issue is still server-side. Re-check permission assignment and intended send mode.

Step 4: Outlook desktop remediation (if OWA works)

If OWA works but Outlook desktop fails, address the client state:

  1. Close Outlook completely.
  2. Wait a short period (permission propagation and token refresh).
  3. Reopen Outlook and test again.
  4. If still failing: remove and re-add the shared mailbox (or create a new Outlook profile for deterministic cache reset).
Engineering note Outlook desktop can hold onto outdated sender resolution and access tokens. A new profile forces a clean resolution path.

If it still fails

  • Confirm the user is selecting the correct From identity (not a similarly named contact or alias).
  • Confirm there are no transport rules or policy restrictions affecting send.
  • Check if the shared mailbox is hidden or represented by a different address in the client.
  • Validate that your organization is not mixing on-premises Exchange delegation with Exchange Online in an inconsistent manner.
