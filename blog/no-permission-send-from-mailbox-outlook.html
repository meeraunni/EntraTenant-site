You don’t have permission to send messages from this mailbox

This is a permission evaluation failure somewhere between Outlook, Exchange delegation, and directory replication. The correct fix is to verify which sender identity Outlook is using, which permission model is required (Send As vs Send on Behalf), and whether the client is operating with stale cached permissions.

2026-03-0310 min readDelegationSend As
Key point: Full Access allows mailbox access. It does not grant send rights. “I can open the mailbox” is not evidence you can send as it.

How Exchange evaluates the sender

When you send email from a shared mailbox, Exchange evaluates the sender identity and the permission type. Send As means the message appears to originate from the shared mailbox and requires Send As permission. Send on Behalf means the message is sent by a user on behalf of the mailbox and requires the mailbox’s on-behalf delegation.

Many incidents come from a mismatch: the helpdesk assigns Full Access and expects Send As, or assigns Send on Behalf but the user selects the shared mailbox address in From, which triggers a Send As evaluation instead.

Why Outlook fails when OWA works

Outlook desktop caches connection state and authorization metadata. If permissions were recently changed, the mailbox was auto-mapped, or the user has multiple profiles/accounts, Outlook can operate with stale authorization data for longer than expected. In those cases, OWA succeeds (server-side evaluation with fresh state) while Outlook fails (cached state).

Validation workflow (what to prove)

First, confirm which From address the client is using. That determines whether you need Send As or Send on Behalf. Second, validate the permission exists at the directory level. Third, validate propagation and client state by testing in OWA. This isolates whether you have a server-side permission gap or a client-side cache/profile issue.

PowerShell verification (Exchange Online)

# Connect-ExchangeOnline

$SharedMailbox = "shared@domain.com"
$User = "user@domain.com"

"Full Access (mailbox access)"
Get-MailboxPermission -Identity $SharedMailbox |
  Where-Object { $_.User -like "*$User*" } |
  Format-Table User,AccessRights,IsInherited -Auto

"Send As (sender identity)"
Get-RecipientPermission -Identity $SharedMailbox |
  Where-Object { $_.Trustee -like "*$User*" } |
  Format-Table Trustee,AccessRights,IsInherited -Auto

"Send on Behalf (delegation)"
(Get-Mailbox -Identity $SharedMailbox).GrantSendOnBehalfTo

Remediation patterns (choose one and standardize)

From an operational perspective, the right move is to standardize your delegation pattern. Decide whether your org uses Send As (preferred for shared identity) or Send on Behalf (preferred when individual attribution is desired), and then configure consistently. Mixing patterns creates user confusion and intermittent failures.

Assigning the permission (Exchange Online PowerShell)

# Send As
Add-RecipientPermission -Identity $SharedMailbox -Trustee $User -AccessRights SendAs -Confirm:$false

# Send on Behalf
Set-Mailbox -Identity $SharedMailbox -GrantSendOnBehalfTo @{Add=$User}

Client stabilization (when it’s Outlook cache)

If OWA works but Outlook fails after permissions are confirmed, treat Outlook as a cache problem. The most reliable fix is to create a new Outlook profile. This forces the client to rebuild mappings and tokens. This is not “turn it off and on,” it is forcing a clean state in a client that is designed to cache aggressively.
